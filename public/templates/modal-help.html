RepoGrams Help

RepoGrams will help you understand a project's history. It uses a simple visual metaphor to represent a project's history across multiple metrics of interest. It supports metrics like:

  1. who made commits, how often, and how much code did their commits change
  2. the number of programming languages used in a project over time
  3. how often developers changed code across different modules (directories)
  4. the use of branches in the repository
  5. and many others!
RepoGrams can help:
  1. managers to analyze a team's performance and pin-point workflow issues
  2. developers to understand the workflow and the history of an unfamiliar software project
  3. researchers in the software engineering community to juxtapose software repositories of multiple projects to select an appropriate set of evaluation targets for a study

How to Use RepoGrams

Grouping the Results

There are two options to choose from when grouping your results. "Group by metric" will group your results by the metrics you selected. For example when you have multiple repositories loaded, it will display the results of each repository under each different metric. "Group by repository" will group the results by repository instead.hi
Selecting a Metric

You can click on the "Switch" button beside "Metric" to enable a pop-up menu with a list of all the available metrics. Each metric will display its corresonding information for each commit in the repository that you have selected. If you did not load a repository yet, RepoGrams will ask you to do so before it displays any diagram.
Selecting a Block Length Mode

There are 3 choices for scaling displayed block widths. The fixed block width will set each commit block width to be the same for each repository. The linear and logarithmic options will scale the width of each commit block linearly and logarithmically respectively according to the number of lines changed in each commit. There are also 2 choices for comparing results between repositories. The "Globally Normalized" block length option will initally set all commit block widths to be the same before the scaling so that results are comparable between repositories. The "Project Normalized" option will optimize the commit block widths for each individual repository so that each result fills the screen.
Helpful Tips

Once the diagram is displayed, you can click on an individual commit to see a pop-up box with the commit ID and it's message as well as the values of each selected metric for that commit. If you click on the commit message, it will take you to the github commit page for that commit.
